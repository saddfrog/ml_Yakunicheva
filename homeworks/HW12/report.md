# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использован датасет `S12-hw-dataset.csv` с почасовыми данными за период с 01.01.2025 по 29.06.2025 (4320 наблюдений). Целевая переменная - `target`.
- Выполнен корректный temporal split: 80% train (первые 3456 наблюдений), 10% val (следующие 432), 10% test (последние 432).
- Сравнены 4 подхода:
  - **B1 (naive-last)**: прогноз = последнему известному значению из train
  - **B2 (moving-average)**: прогноз = скользящее среднее по окну 7
  - **B3 (ridge-lag-features)**: Ridge-регрессия на лаговых (1,7,14), rolling- (mean,std по окнам 7,14) и календарных (час, день недели, выходной) признаках
  - **R1 (gru-forecast)**: GRU с 2 слоями по 64 нейрона, window_size=24 (сутки)
- Лучший подход: **GRU** с валидационной MAE=4.82 и тестовой MAE=5.13, что значительно превосходит все baseline-модели.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Разбиение выполнено строго по времени без перемешивания:
  - Train: 2025-01-01 00:00:00 - 2025-05-24 23:00:00 (3456 наблюдений, 80%)
  - Val: 2025-05-25 00:00:00 - 2025-06-11 23:00:00 (432 наблюдения, 10%)
  - Test: 2025-06-12 00:00:00 - 2025-06-29 23:00:00 (432 наблюдения, 10%)
- Почему random split некорректен: временные ряды имеют автокорреляцию (зависимость от прошлых значений). Random split нарушает временную структуру и приводит к утечке будущих данных в обучение, что даёт завышенную оценку качества. Temporal split сохраняет хронологический порядок и моделирует реальный сценарий прогнозирования.

### 4.2. Признаки для baseline-моделей

- **Лаговые признаки**: `lag_1`, `lag_7`, `lag_14`
- **Rolling-признаки**: `rolling_mean_7`, `rolling_std_7`, `rolling_mean_14`, `rolling_std_14`
- **Календарные признаки**: `hour` (час дня), `day_of_week` (день недели 0-6), `is_weekend` (флаг выходного дня)
- Обработка пропусков: строки с NaN (первые 14 наблюдений после создания признаков) удалены
- Масштабирование: использован `StandardScaler`, обученный только на train (fit на train, transform на val и test)

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (naive-last)**: прогноз = последнему значению из train (95.0)
- **B2 (moving-average)**: прогноз = скользящее среднее по окну 7 (неделя)
- **B3 (ridge-lag-features)**: Ridge-регрессия на признаках, подбор alpha через GridSearch (лучший alpha=0.001)
- **R1 (gru-forecast)**: GRU с 2 слоями по 64 нейрона, dropout=0.2, window_size=24, обучение 50 эпох с early stopping (patience=5)

Дополнительно:
- Основная метрика выбора лучшей модели: **MAE** (средняя абсолютная ошибка)
- Дополнительные метрики: RMSE, MAPE
- `window_size` для GRU: 24 (сутки)
- Batch size: 64
- Максимальное число эпох: 50
- Optimizer: Adam (lr=0.001)
- Learning rate: 0.001
- Как сохранялась лучшая модель: по минимальному val_loss (early stopping)

## 7. Анализ

**Наивный baseline (B1)** оказался самым слабым, как и ожидалось. Прогнозирование постоянным значением не учитывает никакую динамику ряда, что приводит к MAE=31.64. Этот результат служит нижней границей качества.

**Moving-average (B2)** с окном 7 (неделя) показал значительное улучшение (MAE=12.73), так как сглаживает суточные и недельные колебания. Однако он не может предсказывать резкие изменения и экстремальные значения.

**Ridge на признаках (B3)** стал лучшим baseline (MAE=5.15). Лаговые признаки (1,7,14) позволили модели учитывать ближайшие и сезонные зависимости, rolling-статистики добавили информацию о локальных трендах и волатильности, а календарные признаки (час, день недели) уловили суточную и недельную периодичность. Ridge с регуляризацией (alpha=0.001) предотвратил переобучение.

**GRU (R1)** показала наилучший результат (MAE=4.82), что объясняется её способностью:
- Автоматически извлекать временные зависимости из последовательности
- Учитывать долгосрочный контекст через скрытое состояние
- Обучаться на оконном представлении (24 часа), что позволяет модели видеть полный суточный цикл

**Наблюдаемый тренд и сезонность** хорошо видны на графиках: утренний рост (6-9 утра), дневной пик (12-17 часов), ночной спад, выходные дни значительно ниже будних. Все модели, кроме B1, уловили эти паттерны, но GRU сделала это наиболее точно.

**Утечка данных** могла бы возникнуть при использовании random split или при построении признаков с использованием будущих значений. В работе использованы только прошлые значения (shift, rolling) и календарные признаки, что корректно.

**Выбор по validation, а не по test, принципиален**, так как позволяет объективно сравнить модели без использования тестовых данных. Test используется только один раз для финальной оценки лучшей модели, что предотвращает переобучение на тесте.

**Ошибки модели**: наиболее показательные ошибки GRU приходятся на резкие пики (например, 2025-06-12 16:00:00, значение 210.1) и переходные периоды между буднями и выходными. Это указывает на сложность прогнозирования экстремальных событий, что характерно для многих временных рядов.

## 8. Итоговый вывод

- В качестве базового подхода для данной задачи я бы взял **GRU**, так как она:
  - Даёт наилучшее качество (MAE=5.13 на test)
  - Автоматически извлекает временные зависимости
  - Хорошо справляется с суточной и недельной сезонностью
- Простого baseline на признаках (Ridge) достаточно, когда:
  - Временной ряд хорошо описывается линейными зависимостями от прошлых значений
  - Нужна интерпретируемость модели
  - Вычислительные ресурсы ограничены
- Переходить к GRU/другим нейросетям имеет смысл, когда:
  - Ряд имеет сложные нелинейные зависимости
  - Важна высокая точность прогноза
  - Доступны достаточные вычислительные ресурсы
- Главный вывод про корректную валидацию временных рядов: **нельзя использовать random split** — это приводит к утечке будущих данных и нереалистично завышенной оценке качества. Всегда нужно сохранять хронологический порядок и использовать temporal split.
